# Implementation Plan: kpidash MVP — Multi-Client Status Dashboard

**Branch**: `001-mvp-dashboard` | **Date**: 2026-03-31 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-mvp-dashboard/spec.md`

## Summary

kpidash is a fullscreen dashboard running on a Raspberry Pi 5 that displays
real-time health status, system telemetry, activities, repo status, and ambient
fortune content for multiple client machines. This plan covers: (1) refactoring
the existing C/LVGL POC dashboard to use Redis instead of UDP and adding all
MVP widgets; (2) building a cross-platform Python client daemon and CLI; and
(3) building a Python MCP server for AI agent activity reporting.

The transport layer shifts from UDP fire-and-forget to Redis key-value storage,
enabling state persistence, offline detection via TTL, and decoupled read/write
for all components.

## Technical Context

**Dashboard (kpidash binary)**
- **Language/Version**: C11
- **Build System**: CMake 3.22+
- **UI Framework**: LVGL 9.2.2 (DRM/KMS backend — `lv_linux_drm`)
- **Redis Client**: hiredis v1.2.0 (Debian Trixie system package: `libhiredis-dev`)
- **Additional Libs**: cJSON, libpng, libdrm, libfreetype
- **Target Platform**: Raspberry Pi 5, Debian 13 Trixie, aarch64
- **Cross-compile**: x86_64 Ubuntu/Debian → aarch64 via `cmake/aarch64-toolchain.cmake`
- **Testing**: ctest (unit tests where feasible; display/DRM layer tested on-device)
- **Performance Goals**: ~30 fps render; < 10 ms Redis poll cycle on localhost

**Client (kpidash-client Python package)**
- **Language/Version**: Python 3.11+
- **Key Libraries**: psutil, pynvml, redis-py, GitPython, click, tomllib (stdlib)
- **Target Platforms**: Linux (x86_64, aarch64), Windows 10/11
- **Testing**: pytest, ruff (format + lint), ty (type checking)
- **Config format**: TOML (`~/.config/kpidash-client/config.toml`)

**MCP Server (kpidash-mcp Python package)**
- **Language/Version**: Python 3.11+
- **Key Libraries**: mcp (Anthropic MCP SDK), redis-py
- **Target Platforms**: Any machine running an AI agent
- **Testing**: pytest, ruff, ty

**Shared Infrastructure**
- **Storage**: Redis 7.x server on Pi 5 (localhost for dashboard, network for clients)
- **Auth**: `REDISCLI_AUTH` environment variable on all components
- **Scale**: Up to 16 client cards, 10 activities, ~50 repos

**Structure Decision**: Multi-root repository. The C dashboard occupies the
repo root (existing convention). Python packages live under `clients/` to
avoid conflicting with the C `src/` directory specified in the constitution.
The MCP server is a separate package because it has independent dependencies
and deployment lifecycle. Both Python packages are installable via `pip
install -e .` for development.

## Complexity Tracking

| Item | Why Needed | Simpler Alternative Rejected Because |
|------|------------|--------------------------------------|
| Three components (C + Python client + Python MCP) | C required for LVGL; Python required for MCP SDK and faster cross-platform development; MCP server has distinct dependency scope | Single language impossible (LVGL binding friction in Python is significant; C MCP server has immature SDKs) |
| Two Python packages (client + mcp) | Different dependencies, different deployment contexts, different user audiences | One package would mix daemon/CLI concerns with AI-agent-server concerns; creates a heavier install for users who only want the MCP server |
| Redis storage refactor (from POC UDP) | Spec requires persistence, offline detection via TTL, and multi-component communication; UDP is fire-and-forget with no persistence | Keeping UDP would require adding a persistence layer anyway; Redis solves storage + communication in one move |

## Phase 0 Summary — Unknowns Resolved

See `research.md` for full decision rationale. Key resolutions:

1. **Client language**: Python — psutil + pynvml + redis-py + GitPython + click + mcp SDK
2. **C Redis client**: hiredis (Debian Trixie system package, aarch64 available)
3. **Dashboard Redis pattern**: Synchronous 1s poll, non-blocking 1ms timeout
4. **Redis schema**: Per-client JSON strings with TTLs, ZSET for activities, SET for client registry
5. **Config format**: TOML (stdlib tomllib, zero extra Python deps)
6. **Client structure**: One shared cross-platform Python package + separate MCP server
