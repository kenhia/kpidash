# Data Model: kpidash MVP

**Branch**: `001-mvp-dashboard`
**Phase**: 1 (Design)
**Date**: 2026-03-31

This document defines the entities used across the kpidash system: their
fields, relationships, and mapping to Redis storage. The Redis key schema is
the source of truth for inter-process communication; the C structs and Python
dataclasses are derived from it.

---

## Entities

### Client

Represents a reporting machine (or named agent). Created implicitly when a new
hostname first publishes to Redis.

| Field | Type | Source | Notes |
|-------|------|--------|-------|
| `hostname` | string (max 63 chars) | client sends | Unique identifier; used as Redis key segment |
| `priority_order` | int or null | dashboard config | Position in priority client list; null = non-priority |
| `registered_at` | unix timestamp | dashboard infers | Timestamp of first health ping seen |

**Relationships**: Has one `HealthRecord`, one `TelemetrySnapshot`,
zero or more `DiskEntry` (embedded in telemetry), zero or more `RepoStatus`
entries, zero or more `Activity` entries.

---

### HealthRecord

Tracks the liveness of a client. Absence of this key (expired TTL) means the
client is offline.

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `hostname` | string | yes | Key segment |
| `last_seen_ts` | unix float | yes | Set by client on each ping |
| `uptime_seconds` | float | no | Client system uptime; may be absent |

**Redis key**: `kpidash:client:{hostname}:health` (STRING, JSON)
**TTL**: 5 seconds (written by client every 3 s; dashboard checks expiry)
**Offline rule**: Key absent or TTL expired → health indicator red.

---

### TelemetrySnapshot

A point-in-time reading of a client's system resources. Written atomically
as a single JSON string.

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `hostname` | string | yes | Key segment |
| `ts` | unix float | yes | Client-side timestamp of collection |
| `cpu_pct` | float 0–100 | yes | Overall CPU utilisation |
| `top_core_pct` | float 0–100 | yes | Highest single-core utilisation |
| `ram_used_mb` | int | yes | RAM in use (MiB) |
| `ram_total_mb` | int | yes | Total installed RAM (MiB) |
| `gpu` | GpuInfo or null | no | Absent if no GPU reported |
| `disks` | DiskEntry[] | yes | At least one entry if any disks configured |

**Redis key**: `kpidash:client:{hostname}:telemetry` (STRING, JSON)
**TTL**: 15 seconds (written by client every 5 s by default)

---

### GpuInfo

Embedded inside `TelemetrySnapshot`. Only present when client has an NVIDIA GPU.

| Field | Type | Notes |
|-------|------|-------|
| `name` | string | Full model name from NVML, e.g. `"NVIDIA GeForce RTX 4090"` |
| `vram_used_mb` | int | VRAM in use (MiB) |
| `vram_total_mb` | int | Total VRAM (MiB) |
| `compute_pct` | float 0–100 | GPU SM utilisation |

---

### DiskEntry

Embedded list inside `TelemetrySnapshot`. One entry per configured disk.

| Field | Type | Notes |
|-------|------|-------|
| `label` | string | User-defined; displayed on dashboard |
| `type` | enum: `nvme` \| `ssd` \| `hdd` \| `other` | From config (Windows) or auto-detected (Linux) |
| `used_gb` | float | Used space (GiB, 1024-based) |
| `total_gb` | float | Total capacity (GiB) |
| `pct` | float 0–100 | `used_gb / total_gb × 100` |

---

### Activity

A named unit of work. Not tied to a client card — lives in the global
Activities widget. Started and ended by any client (daemon, CLI, or MCP tool).

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `activity_id` | string (UUID4) | yes | Generated by creator at start |
| `host` | string | yes | Reporting hostname or agent identifier |
| `name` | string (max 127 chars) | yes | Human-readable description |
| `state` | enum: `active` \| `done` | yes | `active` on start, `done` on end |
| `start_ts` | unix float | yes | Wall-clock epoch, set by creator |
| `end_ts` | unix float or null | no | Set when state → `done` |

**Redis keys**:
- `kpidash:activity:{activity_id}` — HASH containing all fields above
- `kpidash:activities` — ZSET scored by `start_ts`, member = `activity_id`

**Activity lifecycle**:
1. Creator generates `activity_id` (UUID4).
2. `HSET kpidash:activity:{id} host … name … state active start_ts …`
3. `ZADD kpidash:activities {start_ts} {activity_id}`
4. On end: `HSET kpidash:activity:{id} state done end_ts {ts}`
5. Dashboard reads: `ZREVRANGE kpidash:activities 0 9` → up to 10 IDs →
   `HGETALL kpidash:activity:{id}` for each.
6. Trim: after insert, `ZREMRANGEBYRANK kpidash:activities 0 -21` keeps at
   most 20 in the sorted set (enough history; dashboard shows top 10).

**No TTL** on activity keys — activities persist until explicitly removed by
trim logic or a full Redis flush.

---

### RepoStatus

The git state of a watched repository at the time of last scan. Written per
repo by the client; keyed under the client's hostname.

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `host` | string | yes | Reporting hostname |
| `name` | string | yes | Repo display name (last path segment or user label) |
| `path` | string | yes | Absolute path on the client machine |
| `branch` | string | yes | Current branch name |
| `is_dirty` | bool | yes | True if uncommitted changes exist |
| `ts` | unix float | yes | Timestamp of last scan |

**Redis key**: `kpidash:repos:{hostname}` — HASH, field = `{path}`, value = JSON
**TTL**: 30 seconds (refreshed each scan cycle; absence = client offline/no repos)

Only repos where `branch ≠ default_branch OR is_dirty == true` are written.
Repos returning to clean state are deleted with `HDEL kpidash:repos:{hostname}
{path}`.

---

### FortuneEntry

Text content shown in the Fortune widget. Two independent sources share the
widget: the Pi's `fortune` command (default rotation) and any push from a
client.

| Field | Type | Notes |
|-------|------|-------|
| `text` | string | The fortune text to display |
| `source` | enum: `local` \| `pushed` | Determines display priority |
| `pushed_by` | string or null | Hostname that pushed (if source=pushed) |
| `expires_at` | unix float or null | Only set for pushed fortunes |

**Redis keys**:
- `kpidash:fortune:current` — STRING (JSON). Written by dashboard process on
  each rotation from `fortune` command. No TTL (dashboard owns it).
- `kpidash:fortune:pushed` — STRING (JSON). Written by any client. TTL equals
  the configured display duration (default 300 s). Dashboard checks this key
  first; if present, displays it regardless of rotation timer.

---

### StatusMessage

An operational warning or error raised by the dashboard process itself (e.g.,
widget failure, configuration error, Redis reconnect). Displayed in the status
bar at the bottom of the screen.

| Field | Type | Notes |
|-------|------|-------|
| `message_id` | string (UUID4) | Stable identifier for acknowledgement |
| `severity` | enum: `warning` \| `error` | Controls visual styling |
| `message` | string | Human-readable description |
| `ts` | unix float | When the condition was first recorded |

**Redis keys**:
- `kpidash:status:current` — STRING (JSON). Dashboard writes the currently
  displayed message here. Clients read this to know what is showing.
- `kpidash:status:ack:{message_id}` — STRING. Clients SET this key to
  acknowledge. Dashboard polls for its presence; when found, removes the
  current message (and the ack key) and shows the next queued message or hides
  the status bar.

**Dashboard-internal queue**: The dashboard maintains an in-memory FIFO list
of pending messages. On each Redis poll cycle it checks whether
`kpidash:status:ack:{current_id}` exists; if so, it pops the queue, deletes
the ack key, writes the next message (or DELs `kpidash:status:current` if
the queue is empty).

---

### SystemInfo

Singleton data published by the dashboard process on startup.

| Field | Redis key | Value | Notes |
|-------|-----------|-------|-------|
| Log file path | `kpidash:system:logpath` | STRING | Absolute path on the Pi |
| Dashboard version | `kpidash:system:version` | STRING | Semver string |

Keys have no expiry (dashboard re-writes on each startup). Clients read
`kpidash:system:logpath` via the `log-path` CLI command.

---

### ClientRegistry

Not a data entity but a Redis structure used for client discovery.

| Redis key | Type | Purpose |
|-----------|------|---------|
| `kpidash:clients` | SET | All known hostnames. Clients do `SADD kpidash:clients {hostname}` on startup. Dashboard reads with `SMEMBERS`. No TTL — clients are pruned from display when their health key expires, not from this set. |

## C Struct Mapping (Dashboard)

```c
/* registry.h — extended */
#define MAX_CLIENTS     16
#define HOSTNAME_LEN    64
#define TASK_LEN        128
#define GPU_NAME_LEN    64
#define LABEL_LEN       64
#define MAX_DISKS        8

typedef enum { DISK_NVME, DISK_SSD, DISK_HDD, DISK_OTHER } disk_type_t;

typedef struct {
    char     label[LABEL_LEN];
    disk_type_t type;
    float    used_gb;
    float    total_gb;
    float    pct;
} disk_entry_t;

typedef struct {
    bool      present;
    char      name[GPU_NAME_LEN];
    uint32_t  vram_used_mb;
    uint32_t  vram_total_mb;
    float     compute_pct;
} gpu_info_t;

typedef struct {
    char        hostname[HOSTNAME_LEN];
    bool        online;          /* health key present */
    double      last_seen_ts;
    float       uptime_seconds;
    float       cpu_pct;
    float       top_core_pct;
    uint32_t    ram_used_mb;
    uint32_t    ram_total_mb;
    gpu_info_t  gpu;
    disk_entry_t disks[MAX_DISKS];
    int          disk_count;
    double      telemetry_ts;    /* timestamp from last telemetry write */
} client_info_t;
```

Python dataclasses mirror these fields but use `@dataclass` with Optional
types for nullable GPU fields.

## State Transitions

```
Client health:
  [key present, fresh]  →  online (green indicator)
  [key expiring/absent] →  offline (red indicator)

Activity:
  start()  →  state=active, start_ts set
  end()    →  state=done, end_ts set
  (no cancel concept for MVP)

StatusMessage:
  dashboard raises error  →  written to kpidash:status:current
  client acks             →  ack key written; dashboard removes message
  queue empty             →  kpidash:status:current deleted; bar hidden
```
